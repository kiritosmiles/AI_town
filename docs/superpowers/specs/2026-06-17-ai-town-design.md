# AI Town — LLM 驱动的在线生活模拟游戏 · 设计规格

> 版本: 1.0 | 日期: 2026-06-17 | 状态: 设计阶段

---

## 1. 项目概述

### 1.1 核心概念

AI Town 是一个基于 LLM agent 驱动的多人在线生活模拟游戏。画风与游戏设计参考 RPG Maker（2D 俯视 Tile 地图）。世界定位 21 世纪。每个玩家角色 = 1 个独立 LLM agent，世界内 NPC 也是独立 agent。大脑通过 DeepSeekV4-pro 驱动。

### 1.2 核心差异化

- **真正的 AI 灵魂**: 角色不是脚本驱动，每个行为由 LLM 实时决策
- **人机协作角色扮演**: 玩家设定高层目标，agent 自主规划执行，玩家可随时介入
- **活的世界**: NPC 拥有自己的生活、工作、社交网络，无论玩家在不在，世界都在运转
- **透明化 AI**: 玩家可实时查看 agent 的感知、规划、决策过程

## 2. 技术架构

### 2.1 技术栈

| 层级 | 技术 | 理由 |
|---|---|---|
| 前端 UI | React | 非游戏 UI（面板、菜单、表单），与 React 生态无缝 |
| 游戏渲染 | Phaser.js | 专用 2D 游戏引擎，原生 Tilemap、相机、碰撞、动画支持 |
| 游戏服务器 | Go | goroutine 百万级并发，通道天然适合实时消息广播 |
| Agent 服务 | Python | LLM/AI 生态最优，DeepSeek SDK、prompt 管理、tool 编排 |
| 持久数据库 | PostgreSQL | 角色、建筑、地产、技能、经济、法律日志等结构数据 |
| 实时缓存 | Redis | 在线玩家状态、世界心跳队列、WebSocket 频道订阅、消息队列 |
| 服务间通信 | gRPC (Go ↔ Python) | 高性能、强类型，适合游戏服与 Agent 服之间的结构化调用 |
| 实时通信 | WebSocket (wss) | 浏览器 ↔ Go 游戏服双向低延迟通信 |
| API 网关 | Nginx / Kong | HTTP REST（登录/管理）+ WebSocket 代理 |

```

                         │
                         需要 AI 判断？
                         │
                    ┌────┴────┐
                    │ NO      │ YES
                    │ 直接    │ gRPC → Python Agent 服
                    │ 执行    │         │
                    │         │    LLM 决策 (DeepSeekV4-pro)
                    │         │         │

                    Go 执行行为
                    广播结果 → WebSocket → 所有视野内玩家
```

### 3.2 Agent 意识面板（右侧上）

| 模块 | 数据来源 | 刷新方式 |
|---|---|---|
| 感知 | Agent 从世界状态接收的摘要 | 每次决策周期后推送 |
| 规划 | Agent 的当前目标链 | 目标变更时推送 |
| 行动与后果 | Tool 调用记录 + 执行结果 | 每次行动后实时推送 |

### 3.3 用户意图输入（右侧下）

- 用户输入自由文本作为角色的"首要行动目标"
- 提交后存入 Agent 的长期目标上下文
- Agent 自主规划达成路径，但不违背角色性格
- 优先级可设（首要目标 / 短期愿望 / 仅供参考）
- 示例："尽快攒够 10 万开一家面包店" → Agent 自主分解：报烹饪班 → 加班攒钱 → 关注地价

## 4. 游戏系统

### 4.1 世界地图 — Tile + Plot 双层模型

| 层 | 粒度 | 作用 |
|---|---|---|
| **Tile 层** | 1×1 格，共 1000×1000 = 100 万格 | 地形、行走、视觉渲染、迷雾探索 |
| **Plot 层** | N×N 格（如 10×10），约 1 万个地块 | 地产交易、建造许可、土地用途 |

- **移动**: 角色逐格走，地图按视野渐进加载，远方为迷雾
- **买地**: 在政府大厅（或在线市政系统）查看未售出 Plot，购买
- **建楼**: 在名下 Plot 选择建筑蓝图 → 消耗资源 → 工期 → 建筑呈现
- **未售 Plot**: 系统所有，不可建造但可自由行走

### 4.2 世界生成 — 预设核心 + 生长空间

- 初始上线时中心区域（半径约 100 格）为预设小镇：
  - 政府大楼、商业街、住宅区、公园、学校、医院
  - NPC 均有预设职业家庭和社交关系
- 外围大面积未开发荒野/空地
- 玩家和 NPC 随时间向外扩张建造
- 世界从一个小镇开始"生长"

### 4.3 时间系统 — 实时制

- 24/7 运行，与现实时间连续
- 1 游戏年 ≈ 1 现实月（可调参数）
- 玩家在线时实时交互
- 玩家离线后角色进入"托管模式"（agent 继续运行，行为受限）

### 4.4 经济系统 — 货币 + 材料 + 证书

| 层次 | 内容 | 示例 |
|---|---|---|
| 货币 | 基础交易媒介 | 打工赚现金、购物消费、地产交易 |
| 材料 | 建造/制造所需物理资源 | 木材、石料、钢材、食材 |
| 证书 | 行为资质门槛 | 烹饪证书（开店）、建筑执照（盖高层）、驾照 |

证书通过考试/学习获得，形成职业体系。材料可通过购买、采集、生产获得。

## 5. 角色系统

### 5.1 创建角色 — 引导式问答 + 自由 Prompt

流程：
1. 系统提出引导选择题（性别、年龄段、性格倾向、梦想职业）
2. 用户还需额外填写一段自由 prompt 补充角色背景
3. 系统综合生成角色属性 + 性格参数 + 初始记忆
4. 1 用户 = 1 角色，不可多开

### 5.2 Agent 决策 — 混合模式

关键行动通过 Tool 调用（精确控制），社交/对话通过自然语言意图（自然表达）。

| 类型 | 示例 | 执行方式 |
|---|---|---|
| Tool 调用 | `move_to`, `start_work`, `buy_plot`, `build` | Go 引擎结构执行 |
| 自然语言意图 | 对话、表白、议价、情感表达 | Python Agent 解析 → 传给目标 Agent → LLM 处理 → 返回 |

### 5.3 Tool 学习系统 — 综合型

| 层级 | 解锁方式 | 示例 |
|---|---|---|
| 基础 Tool | 知识获取 — 读书/看视频/观察 | `cook_basic` — 看菜谱即可 |
| 进阶 Tool | 技能熟练 — 反复练习，有失败率 | `cook_advanced` — 基础烹饪成功 N 次后解锁 |
| 特殊 Tool | 社交传授 — 必须由已掌握者"教"给你 | `secret_recipe` — 必须拜师学艺 |

形成"师徒"关系驱动社交。

### 5.4 属性系统

| 类别 | 属性 | 说明 |
|---|---|---|
| 生理 | 健康、饥饿、体力 | 实时消耗，需要吃饭/休息补充 |
| 心理 | 心情、压力、社交满意度 | 影响 agent 决策倾向 |
| 技能 | 各职业技能树 | 影响工作收入和建造能力 |
| 社交 | 声望、人际关系网、好感度 | 影响 NPC 互动和社会机会 |

### 5.5 背包与物品

- 初始容量 20 格
- 可通过背包升级扩大
- 物品类型：消耗品（食物/药）、材料、工具、证书文档、礼物

## 6. 生命与死亡系统

### 6.1 寿命

- 基线寿命: 60-90 岁随机
- 修正因素: 生涯累积健康水平 + 身体条件 + 社会医学发展水平
- 医学研发可延长寿命上限
- 年龄随游戏时间自然增长

### 6.2 死亡

| 死因类型 | 说明 |
|---|---|
| 自然死亡 | 达到寿命上限 |
| 事件死亡 | 疾病、事故、经济破产自杀、他杀 |

### 6.3 死亡后处理

```
角色死亡
  │
  ├── 有成年子嗣？
  │     ├── 是 → 选择 [继承子嗣继续玩] 或 [结束游戏重开]
  │     │         继承: 继续子嗣的人生
  │     │         重开: 子嗣留在世界当 NPC，与新角色完全无关
  │     │
  │     └── 否 → [结束游戏重开]
  │
  └── 死亡结算 → 成就积分
                   │
                   积分可用于新角色初始属性优化（有限度）
```

## 7. 社交与关系系统

### 7.1 婚姻 — 好感度数值 + LLM

- 后台隐藏好感度数值（0-100）
- LLM 聊天自然影响数值（系统量化后调整）
- 数值达标后解锁特定行为：表白 → 恋爱 → 求婚 → 结婚
- 好感度降为 0 时关系破裂，可离婚
- 可与 NPC 或玩家角色恋爱/结婚

### 7.2 生育

- 已婚角色可生育子嗣
- 子嗣继承父母的部分属性
- 子嗣成年（18 游戏岁）后成为可继承对象
- 子嗣也是独立 agent

### 7.3 玩家间社交 — 完全自由 + 可发展法律

- 玩家角色间如同真实世界完全自由交互
- 聊天、交易、组队、PK 均不设系统限制
- 所有社交行为通过 tool 调用 + LLM 自然语言

## 8. 法律与治理系统

### 8.1 混合治理模型

| 层级 | 内容 | 说明 |
|---|---|---|
| NPC 宪法底线 | 硬编码基础法（如恶意杀玩家会被 NPC 警察通缉） | 不可逾越，保证世界稳定 |
| 玩家议会 | 玩家可竞选公职（镇长、议员、法官），参与立法 | 法律可由玩家博弈推动演化 |
| 双重投票 | 重大法律变更需 NPC + 玩家议会双重通过 | 渐进式社会变革 |

### 8.2 执法

- NPC 警察执行法律
- 罪行有对应的处罚（罚款、监禁、流放）

## 9. NPC 系统

### 9.1 NPC 配置

- 纯静态世界：所有 NPC 预先存在，拥有姓名、背景、职业、家庭、社交关系
- 数量：~1000-2000（城镇级）
- NPC 分布在 1000×1000 世界中
- 不论玩家是否在场，NPC 都真实存在并过自己的生活

### 9.2 NPC Agent 运行 — 全量模拟（世界心跳）

- 所有 NPC agent 每 N 分钟跑一轮决策（N 可配，如 5-15 分钟）
- 心跳期间：所有 NPC 做一次批量决策（接下来 N 分钟的行为计划）
- 运行时：Go 执行 NPC 决策行为（移动/工作/消费）
- 玩家与 NPC 交互时：NPC 额外实时响应，不等待心跳
- 玩家视野外 + 无相关关系的 NPC：仅执行心跳决策结果

### 9.3 NPC 生活循环

每个 NPC 拥有：
- 职业（工作地点、工作时间、收入）
- 家庭（住所、家人关系）
- 社交网络（朋友、同事、熟人）
- 消费习惯（购物、娱乐）
- 移动规律（通勤路线）

## 10. 建造系统

### 10.1 蓝图式建造（一期）

- 选择预设建筑蓝图（小木屋、公寓、咖啡店、工厂等）
- 每种蓝图有固定需求：金钱 + 材料 + 工期
- 在自己名下的 Plot 上一键开工
- 计时器走完，建筑出现在 Plot 内的 Tile 上

### 10.2 后续扩展方向（规划中）

- 内部布局自定义（家具摆放、墙纸选择）
- 外观微调
- 多阶段建造（地基 → 框架 → 装修）

## 12. 非功能需求

### 12.1 性能

- 地图分块加载，单屏渲染 20×15 格
- AOI（兴趣区域）算法进行视野广播，避免全量同步
- Go 游戏服无状态化设计，支持水平扩展
- Python Agent 服独立扩缩容

### 12.2 可用性

- WebSocket 断线重连机制
- Agent 决策超时降级（LLM 无响应时回退到预设行为）
- 世界心跳故障隔离（单个 NPC 决策失败不影响全局）

### 12.3 安全

- 用户认证（JWT）
- WebSocket 连接鉴权
- gRPC 内网通信，不对外暴露
